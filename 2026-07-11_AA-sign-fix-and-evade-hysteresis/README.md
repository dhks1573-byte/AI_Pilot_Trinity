# 2026-07-11~13 개선 작업 정리 — AA 공식 부호반전 버그 + Evade 방향떨림 수정

## 한줄 요약

**`AspectAngleUpdate.cpp`의 AA(Aspect Angle) 계산식이 표준 BFM 관례 및 원래
설계 의도와 정반대로 뒤집혀 있었습니다.** ("0=상대 정면"이 되고 있었는데
"0=dead six(상대 꼬리를 정확히 물었을 때)"가 맞습니다.) 이 값은
`BFMClassifier`/`DECO_AspectAngleCheck` 등 전술 판단 로직 전체가 소비하는
핵심 값이라, 한 줄 부호 수정만으로 20시드 배치 기준 `Rule_BFMSelect`

(둘 다 패배 0건 유지). 07-09 스냅샷 이후 가장 큰 개선입니다.

그 위에서 `Task_Evade`의 방향떨림(브레이크 턴 방향이 매틱 뒤집히는 chattering)
버그도 hysteresis로 수정해 **채택**했습니다. 이건 reward만 보면 오히려 악화
(`Rule_BFMSelect` -88.54→-92.24, `Rule_Trinity` -77.73→-94.31)로 보이지만,
대회 규정(07-08 강화된 "무한 회피=패배" 규정, 아래 참고) 관점에서 실제로
측정한 "방어 스트릭 길이"는 뚜렷이 개선되어 채택했습니다. **reward 델타만으로
채택 여부를 판단하면 안 된다**는 이번 작업의 핵심 교훈입니다.

## 변경 1: AA 공식 부호반전 수정 — ✅ 가장 큰 개선

**파일**: `AIP_DCS/BehaviorTree/BT_Content/Service/AspectAngleUpdate.cpp`

기존 코드는 `TFV`(타겟의 Forward Vector)와 `TPM`(타겟→나 벡터)을 그대로
비교해서 "내가 타겟의 기수 방향(정면)에 있을 때 AA=0"이 되고 있었습니다. 이
파일 자체의 원래 주석("AA: 내가 타겟의 꼬리를 얼마나 잘 물었는지")과 표준 BFM
관례(AA=0 → dead six, AA=180 → head-on) 둘 다에 정반대로 뒤집혀 있던 것입니다.
`BFMClassifier`, `DECO_AspectAngleCheck` 등 이 값을 쓰는 모든 소비자가
일관되게 "AA가 작을수록 좋은 위치(상대 뒤)"라고 가정하고 있었기 때문에, 소스
한 곳(`TFV` 부호 반전)만 고치면 모든 소비자가 동시에 올바른 의미를 갖게
됩니다.

```cpp
// 변경 전
float AA = TPM.angleBetween(TFV);

// 변경 후
float AA = TPM.angleBetween(TFV * -1.0f);   // 0=dead six, 180=head-on
```

**검증**: 20시드 배치에서 `Rule_BFMSelect`/`Rule_Trinity` 둘 다 회귀 없이
개선. WEZ 내 평균 ATA도 대략 115도 → 70도 수준으로 크게 개선되어(진단 로그
기준), 조준 자체의 질이 눈에 띄게 좋아졌습니다.

## 변경 2: `Task_Evade` 방향떨림(chattering) 수정 — hysteresis 적용, 채택

**파일**: `AIP_DCS/BehaviorTree/BT_Content/Task/Task_Evade.cpp`

### 원인

`Task_Evade`(방어 브레이크 턴)의 발동 게이트는 `DECO_TailThreatCheck
ThreatAngle=150`, 즉 **타겟이 내 정확한 뒤(dead six 30도 콘) 안에 있을 때만**
발동합니다. 그런데 타겟이 정확히 내 뒤에 있으면 기하학적으로 `Side`(내 오른쪽
벡터와 타겟 방향의 내적)는 **항상 0 근처**로 나옵니다 — 즉 발동 조건 자체가
좌우 판단이 가장 불안정한 지점과 정확히 겹칩니다. 진단 로그로 실측한 결과,
장시간(15~62초) 지속된 방어 스트릭에서 브레이크 방향 부호(`turnSign`)가
8~27% 확률로 매틱 반전하고 있었고, 반전률이 높을수록 스트릭이 길게 지속되는
뚜렷한 상관관계가 확인됐습니다.

### 수정

```cpp
// 변경 전
double Side = MyRight.dot(LOSToTarget);
double TurnSign = (Side >= 0.0) ? 1.0 : -1.0;

// 변경 후
const double SIDE_HYSTERESIS = 0.15;
static double __lastTurnSign[2] = { 1.0, 1.0 };
int __t = ((*BB)->Team == BLUE) ? 0 : 1;
double TurnSign;
if (std::abs(Side) < SIDE_HYSTERESIS)
    TurnSign = __lastTurnSign[__t];   // 확신 없으면 직전 방향 유지
else
    TurnSign = (Side >= 0.0) ? 1.0 : -1.0;
__lastTurnSign[__t] = TurnSign;
```

`|Side| < 0.15`일 때는 새로 판단하지 않고 직전 방향을 유지(latch)해서, 확실히
한쪽으로 넘어갔을 때만 방향을 갱신합니다. 팀별 상태 유지를 위해 이 코드베이스의
기존 `static int __dbg[2]` 패턴과 동일하게 `static double __lastTurnSign[2]`를
사용했습니다.

### 채택 판단 (중요 — 이번 작업 최대 교훈)

이 수정은 20시드 reward 기준으로는 두 룰 다 악화(`Rule_Trinity`는 -16.5점)돼서,
기존 관행이라면 원복 후보였을 것입니다. 하지만 별도로 진행한 대회 규정 조사
(아래 참고)에서 **"reward/승패 지표가 07-08에 강화된 무한회피 규정 리스크를
전혀 반영하지 못한다"**는 근본 결함을 먼저 확인해뒀기 때문에, 이번엔 reward가
아니라 **방어 스트릭 길이를 직접 측정해 채택 기준으로 사용**했습니다:

| 지표 | 수정 전 | 수정 후 |
|---|---|---|
| 15초+ 스트릭 | 8개 | **6개** |
| 30초+ 스트릭 | 5개 | **3개** |
| 최장 스트릭 | 62.0초 | **47.5초** |
| 방어시간 비중 | 15.7% | **9.7%** |
| 순수 Evade 최장 스트릭 | 58.0초 | **11.0초** |
| turnSign 반전률 | 21~27% | **3~5%** |

가설대로 반전률이 급감했고, 그에 따라 순수 회피 스트릭 길이도 극적으로
줄어 원인 규명이 정확했음을 직접 확인했습니다. **이 기준으로는 명백한
개선이므로 채택.** 앞으로 회피/생존 관련 수정은 reward 델타만으로 판단하지
말고 스트릭 측정을 병행할 것을 권장합니다.

## 대회 규정 관련 중요 발견: "무한 회피 = 패배" 규정 리스크 (미해결, 팀 논의 필요)

07-09 스냅샷에서 이미 무한 회피 무승부 규정을 언급했었는데, 디스코드
공지사항을 다시 확인해보니 **07-08에 이 규정이 강화**되어 "한쪽만 일방적으로
회피하고 상대는 추격하면 무승부가 아니라 회피한 쪽의 **패배**"로 바뀌었고
판정 시간도 15초로 단축됐습니다.

**로컬 `termination.py`/배치 검증 하네스에는 이 규정이 전혀 구현되어 있지
않습니다.** 배포 DLL의 `[ACTIVE]` 로그를 8시드 실측 파싱한 결과, 총 17개
방어 스트릭 중 8개가 15초 이상, 5개가 30초 이상 지속됐습니다(최장 62초, 그중

배치 결과 상당수가 실제 대회 규정으로는 패배였을 가능성**이 있습니다.

변경 2(hysteresis)가 이 리스크를 완화하긴 했지만 근본 해결은 아닙니다. 다음

## 조사했지만 "죽은 코드"로 판명된 것: `Controller_CY`의 LOS≥90도 이진 고착

과거(07-06) 세션에서 `Controller_CY::GetStick`의 `PitchCMD`가 `LOS≥90도`일
때 `UpVector.Z` 부호만 보는 이진 커맨드로 바뀌어, 롤이 90/270도를 지날
때마다 부호가 뒤집혀 조준이 고착되는 버그를 규명한 바 있습니다(WEZ 내 ATA가
수렴하지 못하는 근본 원인으로 지목됨). 이번에 4번째로 이 분기를
`cos(UTAngle)` 기준으로 고치는 시도를 했으나, 20시드 배치 + 진단 시나리오
전부 수정 전/후 완전히 동일한 결과가 나왔습니다.

임시 계측을 추가해 확인한 결과, **이 분기가 현재 코드에서 0회 실행**되고
있었습니다. 07-06 이후 이 문제와 별개로 이미 들어간 다른 수정 2건(170도
특이점에서 강제 롤 킥 추가, `RollCMD<0.1` 비대칭 부스트를 `abs(RollCMD)<0.1`로
수정)이 "롤이 어중간하게 걸려 LOS가 90도에 머무는" 전제조건 자체를 이미
없애버린 것으로 보입니다.

**결론: `Controller_CY`의 이 분기를 직접 고치는 접근은 더 이상 가치가
없습니다.** 실측 없이 손대면 "미세한 우연으로 항상 같았다"고 착각하기 쉬우니,
저수준 함수를 고칠 땐 reward 회귀만 보지 말고 분기 진입 여부 자체를 먼저
계측하는 걸 권장합니다. (v4 수정 코드는 순이익이 없어 원복했고, 이 스냅샷에는
포함하지 않았습니다.)

## 남은 문제 (다음 우선순위)

**실사격 조건(ATA 낮은 각) 도달은 여전히 병목이고, 승리는 아직 0건입니다.**
다음 후보:

1. **폐쇄율(closure rate) 관리 강화** — `Task_Pure`의 국소 스로틀보다 훨씬
   이른 단계부터 적극적으로 감속해 500~1000m 구간 체류시간을 늘리는 시도.
2. **`Task_BreakAndReverse`의 REVERSE 단계 지연 해결** — 원인은 규명됨(타겟의
   현재 6시를 쫓는 반응형 목표점이라 30~70초+ 걸리고 거리가 최대 7km까지
   벌어짐), 아직 미수정. 반전 기동 자체가 본질적으로 느린 유형이라 신중 접근
   필요.
3. **일방적 회피=패배 판정을 `termination.py`에 로컬 반영** — 위 "대회 규정
   리스크" 섹션 참고. 앞으로의 회귀 검증을 실제 대회 규칙에 맞게 현실화.
4. `Controller_CY` 직접 수정 방향은 폐기 권장(위 참고).
